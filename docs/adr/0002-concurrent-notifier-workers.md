# ADR-0002: Пул конкурентних воркерів у нотифікаторі

- **Статус:** Прийнято
- **Дата:** 2026-05-08

## Контекст

Після того як сканер виявляє новий реліз, у Redis-чергу потрапляє по одному
завданню (`NotificationJob`) на кожного підтвердженого підписника. Один реліз
популярного репозиторію — це легко десятки тисяч завдань.

Відправлення листа через Mailgun — це HTTP-виклик із мережевою затримкою
(сотні мілісекунд). Якщо обробляти чергу одним воркером послідовно,
10 000 листів = години відправлення. Це повільно і непотрібно — Mailgun
спокійно тримає паралельні запити.

## Рішення

`Notifier` (`internal/service/notifier.go`) запускає **пул воркерів**
(`NOTIFICATION_WORKERS`, типово 10). Кожен воркер у власній горутині
читає чергу через `BRPOP` (`Dequeue`) і незалежно обробляє завдання:

1. Перевіряє дедуп (`IsSent`) — щоб не надсилати дубль.
2. Відправляє лист.
3. Позначає як надісланий (`MarkSent`) — *тільки після* успішної доставки.

Запит на додавання до WaitGroup і коректне завершення по `ctx.Done()`
гарантують, що при зупинці процесу всі воркери виходять чисто.

Воркери не діляться станом між собою — лише через Redis і Mailgun, які
обидва concurrency-safe. Тому додаткових мʼютексів усередині нотифікатора
не треба.

## Наслідки

**Плюси:**
- Пропускна здатність масштабується змінною `NOTIFICATION_WORKERS` без
  змін у коді.
- Повільні листи (тимчасова затримка Mailgun на одному запиті) не
  блокують решту черги — інші воркери продовжують працювати.
- `BRPOP` атомарний: одне завдання дістає рівно один воркер, дублів
  від самої черги бути не може.

**Мінуси:**
- Більше воркерів = більше одночасних зʼєднань до Redis і Mailgun.
  Тому пул Redis (15 зʼєднань) підібрано з запасом для 10 воркерів +
  сканера + API.
- Порядок надсилання листів не гарантується. Для незалежних повідомлень
  про релізи це не проблема.
