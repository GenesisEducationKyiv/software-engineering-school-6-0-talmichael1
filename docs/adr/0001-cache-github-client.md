# ADR-0001: Кешування відповідей GitHub-клієнта в Redis

- **Статус:** Прийнято
- **Дата:** 2026-05-08

## Контекст

Сканер періодично перевіряє кожен репозиторій, на який є хоч один підтверджений
підписник, і викликає `GetLatestRelease` у GitHub API. Якщо на популярний репо
(наприклад, `golang/go`) підписані тисячі людей, ми все одно ходимо в GitHub
лише за одним і тим самим тегом — це той самий запит.

GitHub має жорсткі ліміти: 60 запитів на годину без токена і 5000 з токеном.
Без кешу ми б швидко впиралися в `429`, особливо при коротких інтервалах сканування
або при перевірці існування репозиторію під час підписки.

## Рішення

Загорнули GitHub-клієнт у `CachedGitHubClient` (див. `internal/cache/redis.go`),
який кешує в Redis із TTL **10 хвилин**:

- успішні відповіді (`200`) — для `RepoExists` як сентинел `__exists__`,
  для `GetLatestRelease` як JSON;
- відповіді `404` — як сентинел `__notfound__`, щоб не запитувати зайвий раз GitHub
  через неіснуючі репо;
- помилки `429` і транзієнтні збої **не кешуються** — наступний виклик
  має право спробувати ще раз.

Запис у кеш — best-effort: якщо Redis тимчасово недоступний, помилка ігнорується,
а виклик все одно повертає результат з GitHub.

## Наслідки

**Плюси:**
- 10 000 підписників на один репо = 1 виклик до GitHub за вікно TTL.
- Підписка на популярний репо не йде в GitHub взагалі, якщо хтось інший
  щойно перевірив його існування.
- Затримка до появи нового релізу — не більше 10 хв + інтервал сканування.

**Мінуси:**
- Свіжий реліз може запізнитись на час TTL. Для нотифікацій про релізи
  це прийнятно — це не realtime-система.
- Залежність від Redis. Він уже потрібен як черга, тож додаткового
  компонента ми не вводимо.
