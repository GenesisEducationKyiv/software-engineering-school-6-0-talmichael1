# System Design Record: GitHub Release Notifier

- **Статус:** Прийнято
- **Дата:** 2026-05-08
- **Повʼязані ADR:** [0001](adr/0001-cache-github-client.md), [0002](adr/0002-concurrent-notifier-workers.md), [0003](adr/0003-consumer-defined-interfaces.md)

## Контекст і ціль

Сервіс дозволяє користувачам підписатися на email-сповіщення про нові релізи
GitHub-репозиторіїв. Коли в обраному репо зʼявляється новий тег релізу — усі
підтверджені підписники отримують лист.

**Цілі:**
- Не пропустити жоден реліз (at-least-once доставка).
- Не засипати GitHub запитами (вкластися в ліміт 5000/год з токеном).
- Один процес, який легко задеплоїти на Heroku dyno, перспектива з повним stateless.

**Не цілі:**
- Realtime (затримка до 10–30 хв прийнятна).
- Гарантія exactly-once (дублікати в граничних випадках можливі — їх
  гасить дедуп у Redis).

## Архітектура

**Компоненти:**
- **API** — Gin (HTTP) + gRPC. Обробляє підписки, підтвердження, відписки.
- **Scanner** — пул воркерів, що раз на `SCAN_INTERVAL` (5 хв) обходить
  репо з активними підписками і чекає нові теги релізів.
- **Notifier** — пул воркерів, що читає чергу з Redis і шле листи через Mailgun.
- **Cleanup** — фоновий воркер, який раз на 30 хв чистить непідтверджені
  підписки старші за годину.

## Потік даних: scan → notify

1. Scanner робить SQL JOIN: бере тільки репо, де є хоч один підтверджений підписник.
2. Репо розкидаються по каналу між `SCAN_WORKERS` воркерами.
3. Кожен воркер тягне `GetLatestRelease` через кешований клієнт (див. ADR-0001).
4. Якщо `release.TagName != repo.LastSeenTag` — формуємо `NotificationJob`
   на кожного підписника і пушимо їх у Redis одним пайплайном (`LPUSH`).
5. `last_seen_tag` оновлюємо **тільки після** успішного enqueue — це дає
   at-least-once: якщо процес помре між enqueue і UPDATE, наступний скан
   повторить.
6. Notifier-воркери (див. ADR-0002) читають через `BRPOP`, перевіряють дедуп
   (`EXISTS`), шлють лист, потім позначають `SET` з TTL.

## Сховища

| Що | Де | Чому |
|---|---|---|
| Підписки, репо, токени | PostgreSQL | Реляційні дані, потрібні констрейнти і JOIN. |
| Кеш GitHub-відповідей | Redis (TTL 10 хв) | ADR-0001. |
| Черга нотифікацій | Redis lists (LPUSH/BRPOP) | O(1), не потребує vacuum. |
| Дедуп надісланих листів | Redis (key з TTL) | Швидко, ідемпотентно. |

## Ключові рішення (підсумок)

- **Кеш GitHub** — Redis із TTL 10 хв (ADR-0001).
- **Concurrent воркери нотифікатора** — пул горутин (ADR-0002).
- **Інтерфейси на боці споживача** — кожен сервіс оголошує рівно той
  набір методів, який реально використовує (ADR-0003).
- **Redis як черга, а не таблиця в Postgres** — на масштабі сотень тисяч
  підписників черга-таблиця накопичує мільйони рядків з vacuum-проблемами.
- **Seed `last_seen_tag` при підписці** — щоб новий підписник не отримав
  лист про старий реліз.
- **Rollback при помилці підтвердження email** — інакше користувач застрягне
  з 500 → 409 при ретраї.

## Ризики і компроміси

- **Залежність від Redis.** Якщо він падає — встає і кеш, і черга. Прийнятно:
  на наступному скан-циклі все відбудується (релізи виявляються повторно,
  завдання перевиставляються).
- **Mailgun free tier** — 100 листів на добу. Для прод-демо досить.
- **gRPC на Heroku** — Heroku відкриває один порт на dyno, тому gRPC
  доступний лише в Docker-середовищі.
- **Дублікати листів у крайніх кейсах** — якщо процес помер між
  `email.Send` і `MarkSent`. Свідомо обрано mark-after-send,
  бо втратити лист гірше, ніж надіслати дубль.
